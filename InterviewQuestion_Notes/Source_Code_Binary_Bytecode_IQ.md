# Source Code vs Bytecode vs Binary Code

## Comparison Table

| Aspect | Source Code | Bytecode | Binary Code (Machine Code) |
|---|---|---|---|
| What it is | Human-readable code written by a developer | Intermediate, partially compiled code for a VM/runtime | Pure machine instructions executed directly by the CPU |
| Readability | ✅ Fully readable by humans | ⚠️ Low-level; readable by VMs, not meant for humans | ❌ Not human-readable — just raw 0s and 1s / hex |
| Portability | Cross-platform (runs wherever the runtime exists) | Cross-platform (targets the VM, not a specific CPU) | Platform-specific (tied to a CPU architecture like x86/ARM) |
| Who generates it | Written in an editor (VS Code, etc.) | Generated by the compiler/interpreter (e.g., V8 Ignition) | Generated by the compiler/JIT or assembler |
| Typical file ext. | `.js`, `.py`, `.java`, `.c` | `.class` (Java), `.pyc` (Python), internal VM format | `.exe`, `.o`, `.bin` |
| Execution | Cannot run directly — must be compiled/interpreted | Runs inside a Virtual Machine (JVM, V8, CPython) | Runs directly on the CPU with no intermediate layer |
| Speed | N/A (not executable) | Medium-fast (VM overhead) | Fastest (bare-metal execution) |

## Simple Example Walkthrough

### 1. Source Code

File: `example.js`
```js
const radius = 5;
const area = Math.PI * radius * radius;
console.log(area);
```

This is the source code you write and read. It uses JavaScript syntax, names, and built-in APIs.

### 2. Bytecode

When a runtime like V8 processes the source, it parses it and converts it into bytecode for its interpreter.

Pseudo-bytecode example:
```
LOAD_CONST radius 5
LOAD_GLOBAL Math.PI
MUL radius radius
CALL console.log area
```

The bytecode is not human-source text anymore, but it is still higher-level than raw machine instructions. The runtime uses it to manage control flow, stack values, and object operations.

### 3. Binary Code

The final stage is machine code for the CPU. In V8, the runtime may compile hot bytecode into native instructions.

Pseudo-machine instructions:
```
mov eax, [radius]
fld qword [pi]
fmul qword [eax]
fstp qword [area]
call printf
```

Binary code is the actual bytes loaded into the processor, such as `0x8B 0x45 0xFC` and so on.

## Pipeline Diagram

Source Code -> Parser -> Abstract Syntax Tree -> Bytecode -> JIT Compiler -> Machine Code -> CPU

- Source Code: `example.js`
- Parser: reads text and builds a structure
- AST: abstract syntax tree representing program logic
- Bytecode: VM instructions generated from AST
- JIT Compiler: optionally compiles hot bytecode to machine code
- Machine Code: final instructions executed by CPU

## Why the layers exist

- Source Code is easy for humans to write and maintain.
- Bytecode is optimized for virtual machine execution and portability.
- Binary Code is what the CPU can execute directly.

## TL;DR

Source code is the readable program text you write. Bytecode is an intermediate form produced for a runtime or VM. Binary code is the final machine-level instructions that the CPU executes. The workflow is: write source → compile/parse into bytecode → optionally JIT-compile to binary code for fastest execution.
